0. Extract archive.zip file inside ./dataset

------------------------------------------------------------

1. Basic dataset overview (basic_overview.txt)
The first layer of EDA.
You should know:
- total number of images
- number of classes & class names
- number of images in each split:
  + train (seg_train)
  + test
  + prediction
- number of images in each class inside each split

Why this matter?
- confirms the dataset was loaded correctly
- shows whether class imbalance exists
- helps you describe the dataset formally in the report

------------------------------------------------------------

2. File and data integrity checks (integrity_check.txt)
Before any model work, check whether the dataset is clean.
You should detect:
- corrupted images
- unreadable files
- empty files
- wrong file extensions
- duplicate images
- duplicated filenames
- non-image files accidentally inside folders

Why this matters?
- broken files can crash training code
- duplicates can bias the model
- dirty folders often cause hidden bugs

------------------------------------------------------------

3. Image properties
Even though the dataset says 150×150, you should still verify it.
You should inspect:
- image width and height distribution
- whether all images are truly 150×150 (Yes/No)
- number of channels: grayscale, RGB, RGBA
- data type if loaded as arrays
- file format distribution: jpg, png, jpeg,...

Why this matters?
- preprocessing depends on image consistency
- if some images are grayscale or RGBA, you may need conversion to RGB
- if sizes differ, resizing must be included

------------------------------------------------------------

4. Similarity / duplicate leakage check
This is especially important if train and test are separated already.
You should try to detect:
- exact duplicates within train
- exact duplicates within test
- duplicates across train and test

Why this matters?
- leakage makes model performance look better than it really is
- common issue in image datasets

OTHERS THING WE COULD CONSIDER
------------------------------------------------------------


5. Pixel-level statistics
This is useful for deciding preprocessing.
You should compute:
- mean pixel value per channel (R, G, B)
- standard deviation per channel
- min and max pixel values
- overall pixel intensity histogram
- per-channel histograms

Why this matters?
- helps decide normalization
- shows whether images are too dark, too bright, or low contrast
- useful if you later standardize input using dataset mean/std

Typical report sentence: The dataset-wide RGB mean and standard deviation were computed to support normalization during preprocessing.

------------------------------------------------------------

6. Brightness, contrast, and quality analysis
This is not always mandatory, but it is very useful.
You should examine:
- average brightness per image
- contrast distribution
- blurry images
- very dark or very bright outliers

Why this matters?
- low-quality images may reduce model performance
- shows whether augmentation or filtering is needed

Possible metrics:
- mean grayscale intensity
- variance of Laplacian for blur detection
